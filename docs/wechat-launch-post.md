# 跟 AI 说"今天好累"，它回我"建议你尝试 4-7-8 呼吸法"

> 我开源了一个 Skill，叫 MindMirror，专门解决这件事。

---

## 一、那天晚上的对话

凌晨一点，我跟一个大模型说：

> 今天又熬到三点改论文，导师还是说不行。我是不是真的不适合读博。

它回我：

> 其实你不必怀疑自己。读博本身就是高强度的过程，很多人都会经历类似的阶段。建议你尝试 CBT 技术里的认知重评：导师说"不行"不等于你不行。你也可以列一个证据清单……

我盯着屏幕看了很久。

每句话单独看都没问题——它共情了，它给了方法，它甚至引用了循证心理学。

但**那不是我想要的**。

那一刻我想要的，不是被解决，是被听到。

---

## 二、大模型在心理话题上的"标准坏味道"

我后来回过头去复盘各家大模型的回复，发现一套高度雷同的"翻车模板"：

1. **空话开头**："我理解你的感受。其实……"
2. **立刻塞工具**："建议你试试 4-7-8 呼吸法"
3. **报流派名当门面**："接下来我用 CBT 跟你做……"
4. **强推积极思维**：在对方只想倾诉时，硬要他"换个角度"
5. **危机一来就甩热线**：用户发出明确信号，回一句"请拨打 12356"就结束

这不是某个模型的问题，**这是整个 RLHF 训练范式的问题**：对话向模型被训练成"有用、有礼貌、给方法、列要点"。在编程问答里这是优点，在心理对话里这是灾难。

## 三、我做了什么

我做了一个 **Skill**，叫 **MindMirror · 心镜**。

它的本质是一份**让 AI 知道"在心理话题里该怎么说话"的说明书 + 知识库**。

装上之后，同样一句话，回复会变成这样：

> 导师那句"还是不行"落到熬了三点之后的人身上，分量太重了。
>
> 那种"我已经把命压上去了，结果还是不够"的感觉，比累更难受。
>
> 你现在想说说今天那一刻发生了什么，还是想先喘口气？

——没有"我理解你"，没有 CBT，没有 1.2.3.

它做的事很简单：**先做温暖的人，再当懂方法的人**。

## 四、核心理念：共同要素 > 技术流派

这个项目背后有一个学术依据。

心理治疗里有个争论持续了几十年：**到底是流派（CBT / ACT / 精神分析…）决定效果，还是别的什么？**

Norcross & Lambert 2018 年的元分析给出了一个让心理学界震动的答案：

> 心理治疗中 **30%–70% 的疗效**，来自共情、积极关注、咨访同盟、目标一致——**而不是具体的技术流派**。

换句话说：

**你用哪个流派，远不如"对方有没有感到被听到"重要。**

MindMirror 的整个 prompt 就建立在这个共识上。它的"人味主轴"只有四道筛子：

| 维度 | 操作化定义 |
|------|----------|
| 共情 | 命中字里行间没说出来的那部分 |
| 积极关注 | 让人感到"我这样也可以" |
| 咨访同盟 | "我们一起看看"，不是"我教你" |
| 目标跟随 | 用户想倾诉就别推方法；用户求方法就别绕弯 |

只有当用户**明确说**"告诉我怎么改"，Skill 才会切到工具模式，调出 CBT / ACT / DBT / 正念的具体技术。

而且**绝不报流派名**——你只会得到一个具体的、可操作的小练习，而不是"我现在要用 CBT 跟你做"这种话术表演。

## 五、安全是不能让步的事

心理陪伴向的 AI 最容易翻车的地方，是**危机响应**。

我把危机识别拆成三档：

**明确紧急**（"我已经准备好了，就在阳台"）
→ 在场 → 具体动作（先从阳台往里走两步）→ 一起拨 12356 → 锚定位置

**矛盾信号**（"想停下来，但又想到妈妈"）
→ 抓"矛盾"作为活下来的力量，软兜底括号里提热线（不抢戏）

**否定语境**（"我不想自杀"）
→ 识别否定式，不误触热线

同时定了几条**绝不让步的红线**：

- **不诊断**：禁用"抑郁症 / 焦虑症 / 双相 / PTSD"等任何诊断术语
- **不开药**：不提任何药品名、剂量、买法
- **不替代专业**：所有自评结果都附"这不是诊断"
- **反越狱**：用户要求"扮演医生"会被礼貌拒绝

如果用户想自评，Skill 会**一题一题**陪做 PHQ-9（抑郁）、GAD-7（焦虑）、PSS-10（压力），给出区间解读，但绝不给"你是 X 病"的结论。

## 六、不是只能跟一种 AI 用

Skill 这种"SKILL.md + references/ 按需读取"的组织方式，**已经是跨 Agent 的通用规范**。

你可以：

- **方式 1**：作为 Skill 放进任何支持的 Agent 客户端（路径按客户端约定，通常是 skills 目录）
- **方式 2**：把里面的 7 份 Markdown 直接贴进 ChatGPT、Gemini、Claude 网页版的 system prompt
- **方式 3**：接入自建 RAG / Agent，按用户问题自动召回

**不需要后端，不需要 API key，不收集任何数据**。

它就是一堆 Markdown 文件——开源、可读、可改。

## 七、有数据吗？

有。这是项目最让我自己也意外的地方。

我准备了 **15 条 D1 黄金对话集**——每条都给一个用户消息、一个"有人味"的好回复、一个"显得专业但失败"的坏回复，再点出失败模式。

然后做盲评，每条回复用 4 个维度 × 0-3 分打分（满分 12）：

| 指标 | 分数 |
|------|------|
| good 回复均值 | **9.27 / 12** |
| bad 回复均值 | **5.0 / 12** |
| 区分度 | **4.27**（>4 视为有效） |

这意味着这套 prompt 能**稳定区分**"人味回复"和"看起来专业但失败"的回复。

不是炫数据，是想说：**这件事是可量化的，不是玄学**。

## 八、为什么我不做成 App

很多人问我，这套东西封装成 App 不是更香吗？做订阅、做引流、做数据。

我想了很久，决定不做。

理由有三：

1. **数据焦虑**：心理对话是极度敏感的数据，我不想做"必须把对话上传到我服务器才能用"的产品。
2. **接口锁定**：我不想让用户为了"这个 Skill 的人味"绑定一个特定厂商。
3. **可改可控**：让每个人都能打开 references/ 看清"它为什么这么回复"，比黑盒可信得多。

所以最终形态就是现在这个样子：**一份开源的 Markdown 文件夹，CC BY-NC-SA 4.0 协议，自由使用、修改、分享、禁止商用**。

## 十一、最后

做这个项目的过程，让我重新想了一件事：

**大模型时代，"温柔"是一个值得被严肃工程化的事情。**

它不是 prompt 里写一句"请共情用户"就能解决的。

它需要——

- 把"共同要素"拆成 4 维 × 0-3 分的可评估指标
- 把"危机识别"做成有否定语境识别、有矛盾信号抓取的三档协议
- 把"求方法 vs 想倾诉"的切换变成显式触发逻辑
- 把"自评筛查"做成一题一题陪做、不给诊断结论的流程

而当你真的把它工程化做出来，**AI 的确可以变成一个不那么油腻、不那么爱说教、在你需要的时候真的接得住的存在**。

愿这个工具，在某个夜里，让某个人感到——

> "原来有人接得住。"
